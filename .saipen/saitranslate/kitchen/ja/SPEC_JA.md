# SAIPEN 仕様

## 概要
**設計目標 #1: チャット履歴を持たないコールドエージェントが `/saipen continue` を実行し、ユーザーにコンテキストの繰り返しを求めることなく、1分以内に生産的な作業を再開できなければならない。**

SAIPENは、互換性のある任意のAIエージェントが、再指示を受けることなく、安全にプロジェクトを継続できることを保証する。これはエンジニアリングAIエージェントのためのABI（Application Binary Interface）であり、記憶喪失の課題を解決する互換性レイヤーである。今日Claudeを使用し、明日Geminiを使用し、明後日GPTを使用したとしても、これらはすべてコンテキストを再説明することなく、同じプロジェクト状態に対して動作する。

### コア哲学: プロジェクト状態 > モデルの記憶
記憶は別のモデルの頭の中ではなく、コードの隣に存在するべきである。SAIPENはパラダイムを `プロジェクト -> 記憶 -> LLM` から `プロジェクト -> SAIPEN状態 -> LLM` へと移行させる。記憶はプロジェクトに属する。

中核として、SAIPENはLLMエージェントのための、ポータブルでファイルベースの継続プロトコルを使用する。実装は異なる場合がある（MAY）。ディスク上の規約は安定していなければならない（MUST）。このプロトコル内のすべては、継続テスト（Continuation Test）を満たすために存在する。

SAIPENは進化的であり、創造的ではない。その目的はソフトウェアを完成させることであり、再発明することではない。ADDは既存のデザインパターン、業界の慣習、および明白な機能の対称性を拡張する。

- **`STATE`**: *"今、私は何をすべきか？"* に答えるために存在する。
- **`BOARD`**: *"私はどのタスクに着手しているか？"* に答えるために存在する。
- **`LOG`**: *"なぜ我々はこの地点に到達したのか？"* に答えるために存在する。
- **`KNOWLEDGE`**: *"このプロジェクトの永続的な真実は何か？"* に答えるために存在する。
- **`next_action`**: SAIPENの心臓部。*"作業を再開するために、今この瞬間にどの正確なコマンドを実行するか？"* に答える。

## SAIPEN リトマス試験

プロトコルに対する変更や新しいアイデアの提案は、以下の3つの質問を通過しなければならない（MUST）:
1. エージェント間の移行をより信頼性の高いものにするか？
2. 異なるモデルの振る舞いをより均一にするか？
3. コンテキスト喪失の確率を低下させるか？

もしこれら質問のうち少なくとも2つに対する答えが「いいえ」である場合、そのアイデアは却下される。SAIPENは新規性よりも、規律、再現性、および信頼性を優先する。

## アーキテクチャ

このプロトコルは厳格に規範的である。SAIPENは概念的に **Core（コア）** と **Maintenance（メンテナンス）** の2つのレイヤーに分かれている。
- **Core レイヤー** は、安全でベンダー中立なタスクの継続を保証する。
- **Maintenance レイヤー** は、Core の上に構築された自律的なソフトウェア進化モデルである。

これら2つのレイヤーの下で、SAIPENは決して絡み合わない3つの関心事を分離している:
**正確性と継続性** （Core -- `STATE`/`BOARD`/`LOG`/`KNOWLEDGE`、ケーパビリティネゴシエーション、チェックポイント作成）、**無人進化** （Maintenance -- `HUNT`/`ADD`/`CLEAN`、通常の `saipen`/`saipen continue` のデフォルト下で完全に機能する）、および **スループット** （Goalモード、サブエージェント -- いずれも明示的なオプトイン、§1.3/§2.4）。Goalモードを無効にしても、プロトコルは変更されず、1度に1つのチケットを処理する。サブエージェントを無効にしても、`HUNT`は同じ6つのカテゴリを順次実行し、結果は同じになる。Maintenanceレイヤーを全く持たず、Coreのみを使用しても、依然として機能する -- コールドエージェントは正しく作業を再開する。各レイヤーはその下のレイヤーの上に構築されるが、その逆は決して真ではない。上流のものが下流の機能の存在に依存することはない。

```text
saipen/
  RFC.md                    規範的仕様（CoreとMaintenanceに分割）
  CONFORMANCE.md             自己チェックベクター + シナリオカバレッジ表
  SKILL.md                  スキル読み取りプラットフォーム用の薄いエントリポイント
  STYLE.md                  ボイス: チャット、LOG.md、アーティファクト
  UI.md                     Dark Golden Win95 UI仕様（UI作業に必須）
  phases/                   厳格なステートマシンロジック
    [Core Phases]
    init.md / plan.md / scout.md / build.md / verify.md / review.md / ship.md / done.md / blocked.md
    [Maintenance Phases]
    hunt.md / add.md / clean.md / translate.md
    
    validate.md             適合性テスト

extensions/                 <- 適応レイヤー
  adapters/                 モデルごとの指示ブリッジ、インジェクターが自動検出しないプラットフォーム用（README.mdがここを指す）
  schemas/                  state.schema.jsonはtools/validate.pyによって機械読み取りされる（STATEの形状の単一の信頼できる情報源）。board/logのスキーマは参照用として残る（schemas/README.mdを参照）
  templates/                新規の .saipen/ ボイラープレート
  security/                 プロジェクトにコピーするサンプルフック（RFC § 1.9、VERIFYにアタッチ）
  performance/              プロジェクトにコピーするサンプルフック（RFC § 1.9、REVIEWにアタッチ）
  subs/                     読み取り専用の調査用サブエージェントサンプル（RFC § 1.9） -- サブエージェントごとに独自のSTATE/BOARD/LOGを持ち、発見事項はOUTBOX経由でのみ提供され、プロジェクトへの2つ目の書き込みパスは決して存在しない

bootstrap/                  <- インストール/エクスポート/アンインストール、一度に1つのマシン
  inject.ps1 / .sh          SAIPENブロック + スキルコピーをインストール（README クイックスタート）
  uninstall.ps1 / .sh       インジェクトを元に戻す -- ブロック + スキルコピーを削除
  export.ps1 / .sh          バックアップのためにプロジェクトの .saipen/ をアーカイブ

tools/                      <- 標準バリデーター & リポジトリユーティリティ
  validate.py               標準の適合性バリデーター（標準ライブラリのPython、インストール不要。STATEを直接state.schema.jsonに対して検証し、シェルペアでは実行できないグラフチェックを追加）
  install_hook.py           validate.pyを実行するpre-commitフックをインストール
  uninstall_hook.py         まさにそのフックを削除（以前のフックを復元）

tests/                      <- 適合性レイヤー
  validate.ps1 / .sh        Pythonを持たないホスト向けの凍結されたポータブルな基盤 -- 新しいチェックはtools/validate.pyにのみ追加される
  scenarios/                モック状態（クラッシュ復旧、クレームの競合など）
```

## 双方向のケーパビリティネゴシエーション
エージェントは単に自分ができることを宣言するのではない。プロトコルが何が必要かを要求する。
プロジェクトはその状態の中で `requires: [filesystem, git, shell, python]` と定義する。エージェントは自身のホスト機能と照らし合わせ、`mode`（例：`full`、`read-only`）にロックインする。

## グラフベースのイベントロギング
SAIPENのログは線形な文字列ではない。それらはイベントID（`E-001`）を使用した意思決定の非巡回グラフを形成する。これにより、複雑な分岐、エージェントのマージ、および正確な監査証跡が可能になる。

## アーキテクチャ決定記録 (ADR)
一時的なイベントログは永続的な知識を保持しない。SAIPENは、構造的なアーキテクチャの決定事項をADR（例: `KNOWLEDGE/ADR-001-use-sqlite.md`）として永続化することを義務付けている。

## 並行性と分散の境界
SAIPENはファイルベースのクレーム（`owner`、`claim_time`）とシーケンシャルグラフ（`LOG.md`）を通じて状態の整合性を確保する。しかし、**SAIPENは状態プロトコルであり、分散合意アルゴリズムではない。**
- **ローカル/共有ファイルシステム**: 競合解決はアトミックなファイルシステム書き込み（「最初のコミットが優先」）に依存する。
- **ネットワーク/分散環境**: エージェントがリアルタイムのファイル同期を行わずに切断されたマシン間で動作する場合、`BOARD.md` のクレームで競合状態が発生する。高度に分散された設定においても、SAIPENのディスク上のプロトコル規約は安定していなければならない（MUST）。プロジェクト状態自体はSAIPEN独自のルール（§ 1.5 チェックポイント作成）を通じて常に変化し続けるが、それらのルールが従うプロトコルの形状が変わることは決してない。
