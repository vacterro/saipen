# SAIPENへのコントリビューション

SAIPENは第一に仕様であり、実装は第二のものです。コントリビューションの大部分はアプリケーションコードではなく、`saipen/RFC.md`、`phases/*.md` ファイル、または `tests/` 内の適合性検証ツールの変更となります。

## 変更を提案する前に

あなたのアイデアに対して、[SAIPENリトマステスト](SPEC.md#the-saipen-litmus-test)を実行してください：
1. エージェント間の移行の信頼性を高めるか？
2. 異なるモデルの動作をより均一にするか？
3. コンテキスト消失の確率を低下させるか？

これらの質問のうち、少なくとも2つに対する答えが「いいえ」である場合、そのアイデアが他でどれほど有用であっても、このプロトコルのスコープ外となります。

## ギャップ（不足・問題）を報告する

以下の内容を記載してIssueをオープンしてください：
- ギャップが存在するファイル/セクション（RFC.md、フェーズドキュメント、スキーマ、テストなど）
- 具体的な証拠（引用、コマンドとその出力、現在の動作が壊れるシナリオなど）
- その代わりに期待する動作

「なんとなく違和感がある」といった曖昧な報告は、具体的な `grep`/再現手順に比べて対処が難しくなります。適切な記述形式については、バグレポートのIssueテンプレートを参照してください。

## 変更を加える

1. 編集する前に、`saipen/RFC.md` および関連する `phases/*.md` ファイルを最後まで読んでください。一見ギャップに思える問題の多くは、既に別の場所で対処されているか、ドキュメントに記載された理由により意図的に特定のスコープに制限されています。
2. 先行技術について `CHANGELOG.md` および `.saipen/KNOWLEDGE/decisions.md` を確認してください。過去に決定され却下された事項について、説明なしに再提案しないでください。過去の却下が誤りであったという新しい証拠がある場合は、PRの説明にその旨を明記してください。
3. すべての規範的な変更（MUST/MUST NOT/SHOULD）には `CHANGELOG.md` への追記が必要であり、実用的であれば `tests/validate.sh` および `tests/validate.ps1`（両プラットフォーム）でのカバレッジ、または `tests/scenarios/` 下のフィクスチャが必要です。
4. PRを開く前に両方のバリデータ（検証スクリプト）を実行してください：
   ```bash
   bash tests/validate.sh
   powershell -File tests/validate.ps1
   ```
5. `phases/ship.md` のスキームに従って `VERSION` を上げ（ドキュメントの明確化のみの場合はパッチバージョン、新しい規範的動作の追加はマイナーバージョン、後方互換性のない契約変更はメジャーバージョン）、`README.md` のバージョンバッジと同期させてください。このリポジトリのクローンから実行した場合、`tests/validate.sh`/`.ps1` はこれを自動的にチェックします。

## スタイル

- プロトコルおよびフェーズのドキュメント：簡潔にし、重要な箇所では RFC-2119 のキーワードを使用し、無駄な言葉を省いてください。`saipen/STYLE.md` を参照してください。
- このファイルのすべての内容、コミットメッセージ、コードのコメント、およびCHANGELOG：平易かつプロフェッショナルな表現を使用してください。プロジェクト独自のチャット/LOGボイス（`saipen/STYLE.md`）はアーティファクトには適用されません。

## スコープ外の事項

- SAIPENを分散合意システムに変えること。`SPEC.md` の「Concurrency & Distribution Boundaries」セクションを参照してください。
- 既存のスケルトンを超える、機械解析可能な LOG マーカーの文法。`LOG.md` は固定された形状の周りの散文（テキスト）にとどまります。
- `saipen validate` および `saipen status` と重複するような `saipen doctor` コマンドまたは類似のもの。

これらは過去に提案され、評価されています。これらを再度取り上げるには、単なる再提案ではなく新しい証拠が必要です。
