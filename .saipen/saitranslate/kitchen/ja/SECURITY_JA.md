# セキュリティポリシー

## 適用範囲

SAIPENは仕様であり、インストールおよびエクスポート用の小規模なローカルスクリプト群（`bootstrap/inject.ps1`/`.sh`、`uninstall.ps1`/`.sh`、`export.ps1`/`.sh`）を伴います。サーバーを実行したり、テレメトリを収集したり、どこかにデータを送信したりすることはありません。スクリプトが実行するすべての操作は、ユーザーが既に管理しているファイル（ユーザー自身の`~/.claude`、`~/.gemini`、プロジェクトの`.saipen/`など）に対するローカルファイルシステムへの書き込みのみです。各操作は、最初の変更前に自動的に作成される`.bak`バックアップによって保護されます。

セキュリティレポートの対象として実際に価値があるのは以下の2点です：
1. ブートストラップスクリプトが、自身のコメントやREADMEで説明されている以上のことを、ファイルシステムやgit履歴に対して行っている場合。
2. プロトコル自体のシークレット衛生ルール（RFC.md § 1.1 -- APIキー、トークン、パスワードを`STATE.md`/`BOARD.md`/`LOG.md`/`KNOWLEDGE/`/`kitchen/`に絶対に書き込まないこと）に実際の抜け穴があり、SAIPENに従うエージェントがコミットされるファイルにシークレットを漏洩させてしまう可能性がある場合。

## サポートされるバージョン

`main`ブランチ上の最新のタグ付きリリースのみがサポートされます。これはプロトコル仕様であり、長期稼働するサービスではありません -- LTSブランチは存在しません。

## 脆弱性の報告

GitHub issueを開いてください。レポートが（仮説ではなく）実際に現在悪用可能な問題を含む場合は、修正がリリースされる前に一般に公開されるのを防ぐため、公開のissueの代わりに、このリポジトリの**Security**タブ（"Report a vulnerability"）からプライベート/セキュリティアドバイザリとしてマークしてください。

以下の情報を含めてください：関連するスクリプトまたはRFCルール、具体的なシナリオ、実際に起こることと起こるべきことの比較。他のバグ報告と同じレベルの証拠基準が求められます（`CONTRIBUTING.md`を参照してください）。
