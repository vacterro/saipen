# Специфікація SAIPEN

## Анотація
**Мета дизайну №1: Холодний агент із нульовою історією чату повинен мати можливість виконати `/saipen continue` і відновити продуктивну роботу протягом однієї хвилини, не просячи користувача повторювати контекст.**

SAIPEN гарантує, що будь-який сумісний ШІ агент може безпечно продовжувати будь-який проєкт без повторного інструктажу. Це ABI (Application Binary Interface - двійковий інтерфейс додатків) для інженерних ШІ агентів — шар сумісності, який вирішує проблему амнезії. Незалежно від того, використовуєте ви Claude сьогодні, Gemini завтра, і GPT післязавтра, всі вони працюватимуть з одним і тим самим станом проєкту, не вимагаючи від вас повторювати контекст.

### Основна філософія: Стан проєкту > Пам'ять моделі
Пам'ять повинна жити поруч із кодом, а не всередині голови іншої моделі. SAIPEN зміщує парадигму з `Проєкт -> Пам'ять -> LLM` на `Проєкт -> Стан SAIPEN -> LLM`. Пам'ять належить проєкту.

По суті, SAIPEN використовує портативний, файловий протокол продовження для LLM агентів. Реалізації МОЖУТЬ відрізнятися. Контракт на диску ПОВИНЕН залишатися стабільним. Усе в цьому протоколі існує для обслуговування Тесту Продовження.

SAIPEN є еволюційним, а не креативним. Його мета — завершувати розробку програмного забезпечення, а не винаходити його наново. ADD розширює існуючі шаблони дизайну, галузеві конвенції та очевидну симетрію функцій.

- **`STATE`**: Існує, щоб відповісти *"Що мені робити прямо зараз?"*
- **`BOARD`**: Існує, щоб відповісти *"Яку задачу я беру?"*
- **`LOG`**: Існує, щоб відповісти *"Чому ми дійшли до цього етапу?"*
- **`KNOWLEDGE`**: Існує, щоб відповісти *"Яка довговічна істина цього проєкту?"*
- **`next_action`**: Серце SAIPEN. Він відповідає *"Яку точну команду я маю виконати прямо цієї секунди, щоб відновити роботу?"*

## Лакмусовий папірець SAIPEN

Будь-яка запропонована зміна або нова ідея для протоколу ПОВИННА пройти наступні три питання:
1. Чи робить це перехід між агентами надійнішим?
2. Чи робить це поведінку різних моделей більш однорідною?
3. Чи зменшує це ймовірність втрати контексту?

Якщо відповідь "ні" принаймні на два з цих питань, ідея відхиляється. SAIPEN ставить пріоритет на дисципліні, відтворюваності та надійності вище за новизну.

## Архітектура

Протокол є суворо нормативним. SAIPEN концептуально поділяється на два шари: **Ядро (Core)** та **Обслуговування (Maintenance)**.
- **Шар Ядра** гарантує безпечне продовження задач незалежно від постачальника моделі.
- **Шар Обслуговування** — це автономна модель еволюції ПЗ, побудована поверх Ядра.

Під цими двома шарами SAIPEN розділяє три сфери, які ніколи не переплітаються:
**коректність і продовження** (Ядро -- `STATE`/`BOARD`/`LOG`/`KNOWLEDGE`, узгодження можливостей, 
створення контрольних точок), **автономна еволюція** (Обслуговування -- `HUNT`/`ADD`/`CLEAN`, 
повністю функціональна за замовчуванням при використанні чистого `saipen`/`saipen continue`), та **пропускна здатність**
(Режим Цілі (Goal Mode), Субагенти -- обидва включаються явно, §1.3/§2.4). Вимкніть Goal Mode: протокол 
не зміниться, по одному тікету за раз. Вимкніть Субагентів: `HUNT` запускає ті ж самі шість категорій
послідовно, з однаковим результатом. Використовуйте тільки Ядро, взагалі без шару Обслуговування: воно 
все одно працює -- холодний агент все одно відновлюється коректно. Кожен шар будується на тому, 
що під ним, без зворотного зв'язку; ніщо вище за течією не залежить від існування можливостей
нижче.

```text
saipen/
  RFC.md                    нормативна специфікація (розділена на Ядро та Обслуговування)
  CONFORMANCE.md             вектори самоперевірки + таблиця покриття сценаріїв
  SKILL.md                  тонка точка входу для платформ, що читають навички (skills)
  STYLE.md                  голоси: чат, LOG.md, артефакти
  UI.md                     Специфікація UI Dark Golden Win95 (обов'язкова для робіт з UI)
  phases/                   сувора логіка автомата станів
    [Фази Ядра]
    init.md / plan.md / scout.md / build.md / verify.md / review.md / ship.md / done.md / blocked.md
    [Фази Обслуговування]
    hunt.md / add.md / clean.md / translate.md
    
    validate.md             перевірка відповідності

extensions/                 <- АДАПТИВНИЙ ШАР
  adapters/                 мости інструкцій для кожної моделі, для платформ, які
                             інжектор не визначає автоматично (туди вказує README.md)
  schemas/                  state.schema.json читається програмно інструментом tools/validate.py
                             (єдине джерело істини для форми STATE); схеми board/log
                             залишаються лише довідковими (див. schemas/README.md)
  templates/                свіжий шаблон .saipen/
  security/                 ПРИКЛАД хука для копіювання в проєкт (RFC § 1.9, приєднується до VERIFY)
  performance/              ПРИКЛАД хука для копіювання в проєкт (RFC § 1.9, приєднується до REVIEW)
  subs/                     ПРИКЛАД субагентів тільки для читання та досліджень (RFC § 1.9) -- власні
                             STATE/BOARD/LOG для кожного субагента, знахідки лише через OUTBOX,
                             ніколи не мають другого шляху запису в проєкт

bootstrap/                  <- ВСТАНОВЛЕННЯ/ЕКСПОРТ/ВИДАЛЕННЯ, на одній машині за раз
  inject.ps1 / .sh          встановлює блок SAIPEN + копії навичок (README Швидкий старт)
  uninstall.ps1 / .sh       скасовує встановлення -- видаляє блоки + копії навичок
  export.ps1 / .sh          архівує .saipen/ проєкту для резервного копіювання

tools/                      <- КАНОНІЧНИЙ ВАЛІДАТОР & УТИЛІТИ РЕПОЗИТОРІЮ
  validate.py               канонічний валідатор відповідності (стандартна бібліотека Python, нуль
                             встановлень; валідує STATE проти state.schema.json
                             напряму, плюс перевірки графів, які скрипти shell не можуть зробити)
  install_hook.py           встановлює pre-commit хук, що запускає validate.py
  uninstall_hook.py         видаляє саме цей хук (відновлює будь-який попередній)

tests/                      <- ШАР ВІДПОВІДНОСТІ
  validate.ps1 / .sh        заморожена портативна база для хостів без Python --
                             нові перевірки потрапляють лише в tools/validate.py
  scenarios/                фіктивні стани (відновлення після краху, конфлікти заявок тощо)
```

## Двостороннє узгодження можливостей
Агенти не просто заявляють про те, що вони можуть робити; протокол вимагає того, що потрібно.
Проєкт визначає `requires: [filesystem, git, shell, python]` у своєму стані. Агент звіряє можливості свого хоста з вимогами та фіксується у певному режимі (напр., `full`, `read-only`).

## Журналювання подій на основі графів
Журнали в SAIPEN — це не лінійні рядки. Вони утворюють ациклічний граф рішень за допомогою ідентифікаторів подій (`E-001`). Це дозволяє складні розгалуження, злиття агентів і точні сліди аудиту.

## Записи архітектурних рішень (ADR)
Транзитні журнали подій не зберігають постійних знань. SAIPEN вимагає, щоб структурні архітектурні рішення зберігалися як ADR (напр., `KNOWLEDGE/ADR-001-use-sqlite.md`).

## Межі паралельності та розподілу
SAIPEN забезпечує цілісність стану через файлові заявки (`owner`, `claim_time`) та послідовні графи (`LOG.md`). Однак, **SAIPEN — це протокол стану, а не розподілений алгоритм консенсусу.**
- **Локальна/Спільна файлова система**: Вирішення конфліктів покладається на атомарні записи у файлову систему ("перший коміт перемагає").
- **Мережеві/Розподілені середовища**: Якщо агенти працюють на роз'єднаних машинах без синхронізації файлів у реальному часі, виникатимуть стани гонитви під час заявок у `BOARD.md`. У сильно розподілених середовищах контракт протоколу SAIPEN на диску ПОВИНЕН залишатися стабільним — сам стан проєкту постійно змінюється, через власні правила SAIPEN (§ 1.5 контрольні точки), але форма протоколу, якій підкоряються ці правила, ніколи не змінюється.
