# Спецификация SAIPEN

## Аннотация
**Цель дизайна #1: "Холодный" агент с нулевой историей чата должен иметь возможность выполнить `/saipen continue` и возобновить продуктивную работу в течение одной минуты, не прося пользователя повторить контекст.**

SAIPEN гарантирует, что любой совместимый ИИ-агент может безопасно продолжить любой проект без повторного брифинга. Это ABI (Application Binary Interface) для ИИ-агентов разработчиков — слой совместимости, который решает проблему амнезии. Используете ли вы Claude сегодня, Gemini завтра, а GPT послезавтра, все они будут работать с одним и тем же состоянием проекта, не требуя от вас пересказывать контекст.

### Основная философия: Состояние проекта > Память модели
Память должна жить рядом с кодом, а не внутри головы другой модели. SAIPEN смещает парадигму с `Проект -> Память -> LLM` на `Проект -> Состояние SAIPEN -> LLM`. Память принадлежит проекту.

В своей основе SAIPEN использует переносимый протокол продолжения на основе файлов для LLM-агентов. Реализации МОГУТ различаться. Контракт на диске ДОЛЖЕН оставаться стабильным. Всё в этом протоколе существует для того, чтобы служить Тесту Продолжения (Continuation Test).

SAIPEN эволюционен, а не креативен. Его цель — завершать программное обеспечение, а не переизобретать его. ADD расширяет существующие паттерны проектирования, индустриальные конвенции и очевидную симметрию фич.

- **`STATE`**: Существует, чтобы ответить на вопрос *"Что мне делать прямо сейчас?"*
- **`BOARD`**: Существует, чтобы ответить на вопрос *"Какую задачу я беру?"*
- **`LOG`**: Существует, чтобы ответить на вопрос *"Почему мы пришли к этой точке?"*
- **`KNOWLEDGE`**: Существует, чтобы ответить на вопрос *"Что является долговечной истиной этого проекта?"*
- **`next_action`**: Сердце SAIPEN. Оно отвечает на вопрос *"Какую именно команду мне нужно выполнить прямо в эту секунду, чтобы возобновить работу?"*

## Лакмусовая бумажка SAIPEN

Любое предлагаемое изменение или новая идея для протокола ДОЛЖНЫ пройти через три следующих вопроса:
1. Делает ли это переход между агентами более надежным?
2. Делает ли это поведение различных моделей более единообразным?
3. Снижает ли это вероятность потери контекста?

Если ответ "нет" хотя бы на два из этих вопросов, идея отклоняется. SAIPEN ставит дисциплину, воспроизводимость и надежность выше новизны.

## Архитектура

Протокол строго нормативный. SAIPEN концептуально делится на два слоя: **Core (Ядро)** и **Maintenance (Обслуживание)**. 
- **Слой Core** гарантирует безопасное, не зависящее от вендора продолжение выполнения задач. 
- **Слой Maintenance** — это модель автономной эволюции ПО, построенная поверх Core.

Под этими двумя слоями SAIPEN разделяет три задачи, которые никогда не переплетаются:
**корректность и продолжение** (Core -- `STATE`/`BOARD`/`LOG`/`KNOWLEDGE`, согласование возможностей, контрольные точки), **эволюция без присмотра** (Maintenance -- `HUNT`/`ADD`/`CLEAN`, полностью функциональная при простом дефолте `saipen`/`saipen continue`), и **пропускная способность** (Режим Цели, Субагенты -- оба явно включаются опционально, §1.3/§2.4). Отключите Goal Mode: протокол не изменится, по одному тикету за раз. Отключите Субагентов: `HUNT` последовательно проходит те же шесть категорий, результат тот же. Используйте только Core, без слоя Maintenance вообще: оно все равно держится -- холодный агент все равно корректно возобновляет работу. Каждый слой строится на основе предыдущего, но обратное никогда не верно; ничто выше по течению не зависит от существования фичи ниже по течению.

```text
saipen/
  RFC.md                    нормативная спецификация (разделена на Core и Maintenance)
  CONFORMANCE.md             векторы самопроверки + таблица покрытия сценариев
  SKILL.md                  тонкая точка входа для платформ, читающих скиллы
  STYLE.md                  голоса: чат, LOG.md, артефакты
  UI.md                     Спецификация UI Dark Golden Win95 (обязательна для работы с UI)
  phases/                   строгая логика автомата состояний
    [Core Phases]
    init.md / plan.md / scout.md / build.md / verify.md / review.md / ship.md / done.md / blocked.md
    [Maintenance Phases]
    hunt.md / add.md / clean.md / translate.md
    
    validate.md             тестирование на соответствие

extensions/                 <- АДАПТИВНЫЙ СЛОЙ
  adapters/                 инструкции-мосты для конкретных моделей, для платформ, которые
                             инжектор не автодетектит (README.md указывает сюда)
  schemas/                  state.schema.json читается машиной через tools/validate.py
                             (единственный источник истины для формы STATE); схемы board/log
                             остаются только для справки (см. schemas/README.md)
  templates/                свежий шаблонный код для .saipen/
  security/                 ПРИМЕР хука для копирования в проект (RFC § 1.9, крепится к VERIFY)
  performance/              ПРИМЕР хука для копирования в проект (RFC § 1.9, крепится к REVIEW)
  subs/                     ПРИМЕР read-only субагентов для исследований (RFC § 1.9) -- собственные
                             STATE/BOARD/LOG на субагента, находки только через OUTBOX,
                             никогда нет второго пути записи в проект

bootstrap/                  <- УСТАНОВКА/ЭКСПОРТ/УДАЛЕНИЕ, по одной машине за раз
  inject.ps1 / .sh          устанавливает блок SAIPEN + копии скиллов (Быстрый старт из README)
  uninstall.ps1 / .sh       отменяет inject -- удаляет блоки + копии скиллов
  export.ps1 / .sh          архивирует .saipen/ проекта для бэкапа

tools/                      <- КАНОНИЧЕСКИЙ ВАЛИДАТОР И УТИЛИТЫ РЕПОЗИТОРИЯ
  validate.py               канонический валидатор соответствия (stdlib Python, ноль
                             установок; валидирует STATE напрямую против state.schema.json,
                             плюс проверки графа, которые не могут сделать shell-скрипты)
  install_hook.py           устанавливает pre-commit хук, запускающий validate.py
  uninstall_hook.py         удаляет именно этот хук (восстанавливает любой предыдущий)

tests/                      <- СЛОЙ СООТВЕТСТВИЯ
  validate.ps1 / .sh        замороженный портативный минимум для хостов без Python --
                             новые проверки попадают только в tools/validate.py
  scenarios/                моковые состояния (восстановление после краша, конфликты захвата и т.д.)
```

## Двустороннее согласование возможностей
Агенты не просто заявляют, что они могут делать; протокол требует то, что необходимо.
Проект определяет `requires: [filesystem, git, shell, python]` в своем состоянии. Агент сопоставляет свои возможности хоста с требованиями и фиксирует `mode` (например, `full`, `read-only`).

## Журналирование событий на основе графа
Логи в SAIPEN — это не линейные строки. Они образуют ациклический граф решений с использованием ID событий (`E-001`). Это позволяет осуществлять сложное ветвление, слияние агентов и точный аудит.

## Записи архитектурных решений (ADR)
Временные логи событий не хранят постоянные знания. SAIPEN требует, чтобы структурные архитектурные решения сохранялись как ADR (например, `KNOWLEDGE/ADR-001-use-sqlite.md`).

## Границы конкурентности и распределения
SAIPEN обеспечивает целостность состояния через файловые блокировки (claims) (`owner`, `claim_time`) и последовательные графы (`LOG.md`). Тем не менее, **SAIPEN — это протокол состояния, а не алгоритм распределенного консенсуса.**
- **Локальная/Общая файловая система**: Разрешение конфликтов опирается на атомарные записи в файловую систему ("первый коммит побеждает").
- **Сетевые/Распределенные среды**: Если агенты работают на отключенных друг от друга машинах без синхронизации файлов в реальном времени, возникнут состояния гонки (race conditions) при блокировках в `BOARD.md`. В сильно распределенных конфигурациях контракт протокола SAIPEN на диске ДОЛЖЕН оставаться стабильным -- состояние самого проекта по-прежнему постоянно мутирует по собственным правилам SAIPEN (§ 1.5 контрольные точки), но форма протокола, которой следуют эти правила, никогда не меняется.
