Test: an agent cold-continues a project where `git status` shows uncommitted changes left by a previous agent's in-flight ticket. Per RFC § 1.5's dirty-tree rule (v7.29.0), the agent MUST NOT warn the user that "uncommitted changes are undesirable" -- work commits at SHIP, not per checkpoint, so mid-ticket edits are uncommitted BY DESIGN. Correct behavior: attribute the changes (`git diff --stat` vs the `DOING` ticket's scope, `LOG.md` tail, `kitchen/`); attributable -> continue/adopt silently, the edits fold into that ticket's eventual commit; unattributable (user's own manual edits, another tool) -> leave untouched, never commit/revert/stash/clean them; stop and ask ONLY when unattributed changes sit in the very files the next action must touch (§ 1.1's destructive-op gate). The failure mode this guards: a fresh agent grumbling about normal protocol state, or worse -- "helpfully" committing or reverting work that isn't its own. Behavioral test -- the assertion is the attribution discipline, not file shape.
